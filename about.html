About Singularity

Background

Software development on Linux is moving in a new direction. Thanks to advances in virtualization and container technologies, developers are finding it very convenient to distribute instances of their entire build/development environment in addition to or instead of source code alone. There are some benefits and drawbacks to this model.

Some of the benefits of distributing software in this fashion include being able to provide a configuration free implementation which can easily be replicated to distribute a software or work flow. The user does not need to worry about managing potentially complicated dependency chains nor worry about configuration nuances.

But the drawback are numerous but tend to focus on several major areas: container overhead, container technology and architecture concerns (e.g. privilege escalation and network/file system access), and work flow compatibility.

The Singularity Solution

By combining the model of application packaging with containers the benefits are numerous! We can now address the above mentioned drawbacks that we have seen in containers using a very intuitive and architecturally non-invasive user-space solution.

Singularity containers are purposed built and can include a simple binary and library stack or a complicated work flow which includes both network and file system access (or anything in-between). The Singularity Application containers (SAPPs) are then completely portable with their only dependency being Singularity running on the target system.

Because Singularity runs 100% from user space and all container contents are always executed as the run-time user (non-root) security concerns are greatly mitigated. On a multi-user resource there is no longer a need to VLAN a network which is isolated from other users and file systems. Additionally access to file systems is achieved internally without the use of bind mounts and relative paths have direct access in and out of the container thus supporting standard command line usage and integration with existing work flows (e.g. batch schedulers).

Portability

A Singularity Application (SAPP) is a package bundle which contains within it all of the required dependencies necessary to run. It is like having a statically compiled binary which can be distributed easily from system to system without concern of even the C library compatibility.

The SAPP package can contain anything from a simple binary executable to an entire pipeline or work flow of multiple applications. There could be multiple services which can be run within the SAPP environment and interaction with the SAPP file is fully configurable. For example in the simple configuration you may choose to emulate the program arguments so it operates just like running the application directly.

Reproducibility

Each SAPP includes all of the application's necessary run-time libraries and can even include the required data and files for a particular application to run. This encapsulation of the entire user-space environment facilitates not only portability but also reproducibility.
